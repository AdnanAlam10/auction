# Real-Time Auction App — Technical Specification

A single-auction, room-based live bidding app built to demonstrate real-time full-stack engineering: server-authoritative state, concurrent bid handling, WebSocket sync, and thoughtful domain modelling.

**Timeline target:** 5–7 days for MVP, 1–2 extra days for stretch features.

---

## 1. Product overview

### What it does

A host creates an auction (title, description, starting bid, duration). They get a shareable link. Participants join the link, enter a display name, and bid in real time. Everyone sees the current high bid, the high bidder, the countdown timer, and a live feed of recent bids. When the timer hits zero, the highest bidder wins and everyone sees the result.

### What it deliberately does NOT do

- No user accounts, email, or passwords (display names only)
- No payment processing or real money
- No images beyond an optional image URL
- No auction listings page or search (one auction per room, accessed by link)
- No categories, tags, or filtering
- No bid history beyond the last 10 visible bids
- No chat
- No admin panel

Scope discipline is the most important thing in this project. Every feature outside the MVP list below is a trap.

### The story you're telling on your resume

"Built a real-time auction platform with server-authoritative state, concurrent bid handling using Postgres row-level locks, and an anti-sniping timer extension inspired by eBay. Deployed on Render with Socket.io for bidirectional communication."

## 2. Tech stack

### Frontend
- **React 18** with **TypeScript**
- **Vite** for the build tool (faster and simpler than CRA)
- **Tailwind CSS** for styling (no component library — keep it custom but simple)
- **Socket.io client** for WebSocket connection
- **React Router** for two routes: `/` (create auction / landing) and `/auction/:id`
- **date-fns** for timer formatting

### Backend
- **Node.js** (20+)
- **Express** with **TypeScript**
- **Socket.io** server
- **Prisma** as the ORM (genuinely worth it for speed of development; raw SQL is fine if you prefer but Prisma will save you hours)
- **zod** for request validation

### Database
- **PostgreSQL 15+**
- Local: Postgres via Docker or a local install
- Production: Render or Railway managed Postgres (both have free tiers)

### Deployment
- **Render** (recommended) — supports WebSockets on the free tier, includes managed Postgres, single-service deploy from GitHub
- Alternative: Railway (similar, slightly different UX)
- Do NOT use Vercel for this project — their serverless functions don't support long-lived WebSocket connections well

### Why these choices specifically
- TypeScript everywhere gives you type safety across the wire (you can share types between frontend and backend)
- Socket.io over raw WebSockets because it handles reconnection, rooms, and fallbacks — all things you'd otherwise have to build yourself
- Prisma because auction logic needs transactions, and Prisma's transaction API is clean and typed
- Render because it's the simplest path from "code on GitHub" to "running on the internet with WebSockets working"

## 3. Data model

### Tables

**`auctions`**

| Column              | Type          | Notes                                                    |
|---------------------|---------------|----------------------------------------------------------|
| `id`                | `uuid`        | Primary key, generated by DB                             |
| `title`             | `varchar(200)`| Required                                                 |
| `description`       | `text`        | Optional                                                 |
| `image_url`         | `text`        | Optional, validated as URL                               |
| `starting_bid`      | `integer`     | In cents. Minimum 100 (one dollar)                       |
| `current_bid`       | `integer`     | In cents. Starts equal to `starting_bid`                 |
| `current_bidder`    | `varchar(50)` | Display name of high bidder (cosmetic), nullable         |
| `current_bidder_id` | `uuid`        | Stable per-browser participant id of high bidder, nullable |
| `min_increment`     | `integer`     | In cents. Default 100 (one dollar)                       |
| `starts_at`         | `timestamptz` | When the auction goes live                               |
| `ends_at`           | `timestamptz` | When it ends (mutable — anti-sniping changes this)       |
| `status`            | `enum`        | `pending` \| `active` \| `ended` (Postgres/Prisma enum)  |
| `host_token`        | `uuid`        | Secret — only the host has this, used for host actions   |
| `winner_name`       | `varchar(50)` | Set when auction ends, nullable if no bids               |
| `winner_amount`     | `integer`     | Set when auction ends, nullable                          |
| `created_at`        | `timestamptz` | Default now()                                            |

**`bids`**

| Column        | Type          | Notes                                                    |
|---------------|---------------|----------------------------------------------------------|
| `id`          | `uuid`        | Primary key                                              |
| `auction_id`  | `uuid`        | Foreign key to `auctions.id`, indexed                    |
| `bidder_name` | `varchar(50)` | Display name (cosmetic)                                  |
| `bidder_id`   | `uuid`        | Stable per-browser participant id, indexed              |
| `amount`      | `integer`     | In cents                                                 |
| `created_at`  | `timestamptz` | Default now(), indexed                                   |

### Key design decisions

**Money in cents, not decimals.** Always store money as integers. This prevents floating-point bugs and is a signal of engineering maturity when interviewers look at your schema.

**`host_token` as a separate secret.** The auction `id` is shareable (in the URL), but the `host_token` is returned *only once* on creation and stored in the host's browser localStorage. Host-only actions (cancelling the auction, for example) require the token. This is a lightweight auth pattern that avoids building real auth.

**`current_bid` denormalized on the auction row.** Yes, you could compute it from `max(bids.amount)`, but keeping it on the auction row means you can update it in the same transaction as the bid insert and use it for the concurrency check. This is deliberate denormalisation for correctness and performance.

**`ends_at` is mutable.** The anti-sniping rule extends it. Store the current end time, not the original duration.

**Participant identity is a UUID, not a display name.** Display names are cosmetic and can collide ("Alice" and "Alice" are two different humans). Each browser generates a `participantId` (uuid) on first visit and stores it in `localStorage`. It is sent with every `join_auction` and `place_bid` and used server-side for identity checks (e.g. "don't let the current high bidder bid against themselves"). This is not authentication — it's a stable per-browser handle. Names remain for display only.

## 4. The concurrency problem (the important part)

This is the single most important piece of engineering in the whole project. Get this right and you have a real story to tell in interviews. Get it wrong and you have a racey todo list with a timer.

### The problem

Two users click "Bid $50" at the same millisecond. Without protection, both reads of `current_bid` see `$45`, both validate "$50 > $45 ✓", and both inserts succeed. Now you have two "winning" bids at $50 and the auction state is corrupted.

### The solution: row-level locking in a transaction

When a bid comes in, the backend does this **inside a single Postgres transaction**:

1. `SELECT ... FROM auctions WHERE id = $1 FOR UPDATE` — this locks the auction row until the transaction commits or rolls back. Any other transaction trying to do the same `SELECT FOR UPDATE` will wait.
2. Validate the bid amount against the now-locked `current_bid`:
   - Auction status must be `active`
   - `now()` must be before `ends_at`
   - `amount >= current_bid + min_increment`
   - `bidder_id` is not the same as `current_bidder_id` (prevents the high bidder from bidding against themselves — compared by participant id, not display name, so name collisions don't matter)
3. If valid: `INSERT INTO bids` and `UPDATE auctions SET current_bid = $1, current_bidder = $2`
4. Commit the transaction
5. Only AFTER commit, broadcast the new bid to the room via Socket.io

If any step fails, roll back and emit a `bid_rejected` event to the one client that bid, with the reason.

### Prisma implementation sketch

```typescript
const { newEndsAt } = await prisma.$transaction(async (tx) => {
  // Lock the row
  const auction = await tx.$queryRaw<Auction[]>`
    SELECT * FROM auctions WHERE id = ${auctionId} FOR UPDATE
  `;

  if (!auction[0]) throw new Error('AUCTION_NOT_FOUND');
  if (auction[0].status !== 'active') throw new Error('AUCTION_NOT_ACTIVE');
  if (new Date() > auction[0].ends_at) throw new Error('AUCTION_ENDED');
  if (amount < auction[0].current_bid + auction[0].min_increment) {
    throw new Error('BID_TOO_LOW');
  }

  // Anti-sniping: if within final 30s, push ends_at out.
  // IMPORTANT: this UPDATE happens inside the same transaction as the bid.
  let newEndsAt: Date | null = null;
  const msRemaining = auction[0].ends_at.getTime() - Date.now();
  if (msRemaining <= 30_000) {
    newEndsAt = new Date(Date.now() + 30_000);
  }

  await tx.bid.create({
    data: { auctionId, bidderId, bidderName, amount },
  });

  await tx.auction.update({
    where: { id: auctionId },
    data: {
      currentBid: amount,
      currentBidder: bidderName,
      currentBidderId: bidderId,
      ...(newEndsAt ? { endsAt: newEndsAt } : {}),
    },
  });

  return { newEndsAt };
});
// Only broadcast — and only touch the in-memory timer map — AFTER commit.
io.to(`auction:${auctionId}`).emit('new_bid', { ... });
if (newEndsAt) {
  rescheduleAuctionEnd(auctionId, newEndsAt); // clears old setTimeout, schedules new one
  io.to(`auction:${auctionId}`).emit('timer_extended', { newEndsAt });
}
```

**Read this before you start:** Postgres documentation on `SELECT FOR UPDATE`, and one article on Prisma transactions. Thirty minutes upfront, saves a day of debugging.

## 5. Timer authority

### The rule

**The server is the only authority on time.** The client displays a countdown, but the countdown is computed from `ends_at` received from the server. The server, not the client, decides when the auction ends.

### Implementation

When an auction becomes `active`, the server schedules a `setTimeout` to fire at `ends_at`. When it fires:

1. Open a transaction
2. Lock the auction row (`FOR UPDATE`)
3. Set `status = 'ended'`
4. Determine the winner (the current `current_bidder` and `current_bid`, or null if no bids)
5. Write `winner_name` and `winner_amount`
6. Commit
7. Broadcast `auction_ended` to the room

### Anti-sniping ordering

If a bid arrives in the final 30 seconds, `ends_at` is pushed to `now() + 30s`. The ordering matters — do it wrong and you get timers firing against stale state, or a timer that was scheduled against an `ends_at` that never got committed:

1. Open the bid transaction, lock the auction row with `SELECT ... FOR UPDATE`
2. Validate the bid as normal
3. If this bid falls in the anti-sniping window, include the new `ends_at` in the same `UPDATE` that writes `current_bid` and `current_bidder` — **inside** the transaction
4. Commit
5. **After commit:** look up the existing `setTimeout` handle in the in-memory `Map<auctionId, NodeJS.Timeout>`, clear it, and schedule a new one against the new `ends_at`
6. Emit `new_bid` (and `timer_extended` if the end time changed) to the room

Never clear or reschedule the timer inside the transaction callback. If the transaction rolls back, you've already mutated the timer state and can't undo it. The in-memory map is only mutated from post-commit code paths and from the `auction_ended` firing itself.

**Caveat for honesty:** this in-memory approach means if your server restarts, scheduled timeouts are lost. For a portfolio project this is fine — mention it in the README as a known limitation and say "in production I'd use a job queue like BullMQ or a cron sweep." That caveat itself is a positive signal.

### Client-side countdown

The client receives `ends_at` as an ISO timestamp. It displays a countdown by computing `ends_at - Date.now()` and re-rendering every 100ms (not every second — smoother UX). It also listens for `timer_extended` events and updates its local `ends_at` when they arrive. The client never decides the auction is over — it waits for the `auction_ended` event from the server.

## 6. WebSocket event contract

### Rooms

Every auction is a Socket.io room named `auction:<uuid>`. When a client joins, they join that room. All auction-scoped events are emitted to that room.

### Client → Server events

| Event             | Payload                                                                    | Purpose                                          |
|-------------------|----------------------------------------------------------------------------|--------------------------------------------------|
| `join_auction`    | `{ auctionId: string, displayName: string, participantId: string }`        | Join the room and get initial state              |
| `place_bid`       | `{ auctionId: string, amount: number, participantId: string }`             | Submit a bid                                     |
| `leave_auction`   | `{ auctionId: string }`                                                    | Explicit leave (also handled on disconnect)      |

On `join_auction`, the server stashes `{ participantId, displayName }` on `socket.data` so subsequent events can be attributed without re-sending. `place_bid` still includes `participantId` in the payload for defence-in-depth (validated against `socket.data`). If the two disagree, reject the bid.

### Server → Client events

| Event                    | Payload                                                             | Emitted to            |
|--------------------------|---------------------------------------------------------------------|-----------------------|
| `auction_state`          | Full auction object + recent bids + participant count               | Joining socket only   |
| `new_bid`                | `{ bidderName, amount, currentBid, currentBidder, timestamp }`      | Whole room            |
| `bid_rejected`           | `{ reason: BidRejectReason }` — see exhaustive enum below           | Rejected socket only  |
| `participant_count`      | `{ count: number }` — number of sockets currently in the room       | Whole room            |
| `timer_extended`         | `{ newEndsAt: string }`                                             | Whole room            |
| `auction_ended`          | `{ winnerName: string \| null, winnerAmount: number \| null }`      | Whole room            |

### Bid rejection reasons (exhaustive enum)

```typescript
import { z } from 'zod';

export const BidRejectReason = z.enum([
  'BID_TOO_LOW',
  'AUCTION_NOT_ACTIVE',
  'AUCTION_ENDED',
  'SELF_BID',          // already the current high bidder
  'RATE_LIMITED',      // bids per socket throttled
  'INVALID_AMOUNT',    // not a positive integer
  'NOT_IN_ROOM',       // place_bid arrived before join_auction
]);
export type BidRejectReason = z.infer<typeof BidRejectReason>;
```

Keep this list exhaustive and shared via `shared/types.ts` (see §10). The client renders a message per reason with a `switch`, so adding a reason on the server without updating the client is a type error.

**Internal → wire mapping.** `bidService` throws internal error tokens that the socket handler catches and maps to a `BidRejectReason` before emitting `bid_rejected`. The non-obvious one:

| Internal throw        | Wire reason      | Notes                                                                  |
|-----------------------|------------------|------------------------------------------------------------------------|
| `AUCTION_NOT_FOUND`   | `NOT_IN_ROOM`    | Shouldn't happen in practice — `join_auction` validates existence first. If it does, the client's room membership is inconsistent with the DB, so `NOT_IN_ROOM` is the honest message to show |
| `AUCTION_NOT_ACTIVE`  | `AUCTION_NOT_ACTIVE` | Direct                                                                 |
| `AUCTION_ENDED`       | `AUCTION_ENDED`  | Direct                                                                 |
| `BID_TOO_LOW`         | `BID_TOO_LOW`    | Direct                                                                 |
| `SELF_BID`            | `SELF_BID`       | Direct                                                                 |

### Participant count semantics

`participant_count` is the number of sockets currently joined to `auction:<id>`, computed from `io.sockets.adapter.rooms.get(room)?.size ?? 0`. A single user with two tabs counts as two. This is intentional — it's cheap, predictable, and doesn't require tracking participant identity across reconnects. Broadcast it on every join and disconnect event within the room.

### Validation

Every incoming event MUST be validated with zod before being processed. Never trust the client. Specifically:
- `amount` must be a positive integer
- `displayName` must be 1–50 chars, trimmed, non-empty
- `auctionId` must be a valid UUID
- `participantId` must be a valid UUID

## 7. REST endpoints

Keep REST minimal — most interaction is over WebSockets. REST is only for things that happen before/outside a live session.

| Method | Path                    | Purpose                                                     |
|--------|-------------------------|-------------------------------------------------------------|
| `POST` | `/api/auctions`         | Create an auction, returns `{ id, hostToken }`              |

That's it. Everything else — including the initial render of `/auction/:id` — happens over the WebSocket. The page mounts a skeleton, connects the socket, emits `join_auction`, and populates from the `auction_state` event. Having a single source of truth (the socket) eliminates "REST says X, socket says Y" staleness bugs.

### POST /api/auctions request body

```json
{
  "title": "Vintage camera",
  "description": "Optional text",
  "imageUrl": "https://...",
  "startingBid": 1000,
  "minIncrement": 100,
  "durationSeconds": 300
}
```

The server computes `starts_at = now()` and `ends_at = now() + durationSeconds`, sets `status = 'active'`, generates a `host_token`, and returns:

```json
{
  "id": "uuid",
  "hostToken": "uuid"
}
```

For MVP, auctions start immediately on creation. No scheduled auctions.

## 8. Frontend pages and flows

### Route: `/` — landing / create auction

A single form:
- Title (required)
- Description (optional textarea)
- Image URL (optional)
- Starting bid in dollars (required, converted to cents before sending)
- Minimum increment in dollars (default $1)
- Duration in minutes (dropdown: 1, 2, 5, 10, 30)
- Submit button: "Start auction"

On submit: POST to `/api/auctions`, store `hostToken` in localStorage keyed by auction id, navigate to `/auction/:id`.

### Route: `/auction/:id` — the live auction view

On mount, the page renders a skeleton, reads `participantId` from `localStorage` (generating one with `crypto.randomUUID()` if missing), opens the Socket.io connection, and emits `join_auction`. It fills in from the first `auction_state` event. There is no REST fetch on load — the socket is the only source of truth.

Two modes based on whether the user is the host (has the `hostToken` in localStorage for this auction id) or a participant. The layout is mostly the same; the host sees an extra section.

**Top section — auction info:**
- Title, description, image
- Current bid (large, prominent)
- Current bidder name (or "No bids yet")
- Time remaining (large countdown, updates every 100ms)
- Participant count ("5 people watching")

**Middle section — bid form (participants only):**
- First visit: prompt for display name, store in localStorage
- Input: "Your bid ($)" — pre-filled with the minimum valid bid
- Submit button: "Place bid"
- Shows inline error if the server rejects the bid
- Disabled if auction has ended

**Host section (host only):**
- "You are the host" banner
- Share link with copy button
- (Optional stretch: cancel auction button)

**Recent bids section:**
- Last 10 bids, newest at top
- Animate new bids in (small slide/fade)
- Show bidder name, amount, and relative time ("2 seconds ago")

**Ended state:**
- When `auction_ended` arrives, replace the bid form with a winner banner:
  - "Auction ended — Winner: Alice at $52" or "Auction ended — no bids"
- Disable bidding, stop the countdown at 0

### Timer visual treatment

- Normal: large neutral-coloured countdown
- Last 30 seconds: turn the countdown orange
- Last 10 seconds: turn red and pulse
- When `timer_extended` arrives in the final 30 seconds: brief flash/animation with "Timer extended!" toast

These details are what make the demo feel alive in an interview. Don't skip them.

It's ~8 lines in the `$transaction` callback plus the post-commit reschedule — and it's one of the best signals in the whole project, so it belongs in MVP, not stretch. ### Stretch tier 2 (only if everything above is done)

## 12. Testing strategy

You don't need comprehensive tests for a portfolio project, but you should have **at least one test that proves the concurrency fix works**. This is the thing an interviewer might ask about, and having an actual test is much more convincing than "yeah I handled it."

### Minimum viable tests

1. **Concurrency test (backend).** Spin up 20 concurrent bid attempts at the same amount, verify exactly one succeeds and the others are rejected. Use Vitest or Jest with a test database.
2. **Bid validation unit tests.** Cover the obvious cases: bid too low, auction ended, auction not active.
3. **One E2E smoke test (optional).** Playwright spinning up two browser contexts, creating an auction in one, bidding from the other, verifying the first sees the update. This is the flex test if you have time.

Skip frontend unit tests entirely. Not worth the time for this project.

   - The anti-sniping rule and the post-commit ordering that makes it safe

## 14. Things that will trip you up (warnings)

- **Socket.io rooms.** Make sure to `socket.join('auction:' + id)` on the join event, not just track it in your own state. `io.to(room).emit()` only works if you actually joined the Socket.io room.
- **Transaction scope with Prisma.** Don't put the socket `emit` *inside* the transaction callback — if the transaction rolls back, you've already broadcast. Emit only after `prisma.$transaction` resolves.
- **Timer cleanup on server restart.** In development with hot reload, your in-memory timeout map gets wiped every restart. This is fine. Just know that it happens and don't panic.
- **CORS in production.** When you deploy, the frontend and backend are on different domains. Configure Socket.io and Express CORS explicitly. Test this on the deployed version, not just locally.
- **Money as floats.** Never. Cents as integers everywhere. Convert to dollars only in the UI layer.
- **Timer drift on the client.** Don't use `setInterval(fn, 1000)` for the countdown — it drifts. Use `setInterval(fn, 100)` and recompute from `ends_at - Date.now()` each tick.
- **Reconnect edge cases.** When a socket reconnects mid-auction, it needs to re-join the room and re-fetch state. Socket.io doesn't do this automatically — wire up a `reconnect` handler that re-emits `join_auction`.
- **"Bid too low" race condition in the UI.** The minimum valid bid the user sees might be stale by the time they submit. That's fine — the server will reject it and the UI will show the error. Don't try to prevent this client-side, just handle the rejection gracefully.

## 15. Definition of "done" for MVP

You can call the MVP done when all of these are true:

- [ ] You can create an auction via the form and get a shareable URL
- [ ] Two browser windows can join the same auction and see each other's bids in real time
- [ ] A bid lower than `current_bid + min_increment` is rejected with a clear error
- [ ] The countdown is smooth and accurate
- [ ] The auction ends automatically and announces a winner to both windows
- [ ] You've tested concurrent bids and exactly one wins each round
- [ ] The app is deployed at a real URL with HTTPS
- [ ] The README has the "Key engineering decisions" section written
- [ ] There's a 60-second demo video linked in the README

If you hit all of those, you have a portfolio project that's genuinely better than what most junior applicants submit. Anything beyond that is bonus.

## 16. What to do *before* writing any code

Spend 2 hours on these. They'll save you days:

1. **Read the Postgres docs on `SELECT FOR UPDATE`** — specifically the "Locking" section. You need to understand what it locks and what it doesn't.
2. **Read one article on Prisma interactive transactions** (`prisma.$transaction` with a callback). Understand the difference between interactive and sequential transactions.
3. **Read the Socket.io "Rooms" documentation.** Five minutes, saves you hours.
4. **Sketch the architecture diagram on paper first.** Not because you need a plan — because the act of drawing it will surface questions you haven't thought of yet.
5. **Write the README stub first** — just the sections with placeholders. You'll fill it in as you build, but having the skeleton there keeps you honest about what you're building and why.

That's the spec. Build it in order, resist scope creep, and when in doubt, re-read section 9 and ask yourself if the feature you're about to add is on the MVP list. If it isn't, it can wait.
